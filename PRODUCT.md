# Product

## Register

product

## Users

GAULA unit operators and administrators who staff Colombia's Line 147 extortion and kidnapping hotline. Operators work in high-pressure, time-sensitive contexts: receiving reports, classifying urgency, and routing cases. Admins need a command-level view of all active reports, distribution by type, and operational flow status. Both roles are institutional law enforcement personnel who expect the tool to be fast, unambiguous, and reliable, not pretty.

## Product Purpose

NEXO-147 is a demo platform for receiving, classifying, and tracking initial reports through the GAULA Line 147 anti-extortion system. It provides a structured intake form for operators, a real-time command dashboard for admins, and a role-based access model that enforces information compartmentalization. Success means a GAULA unit can go from "call received" to "report logged and routed" in under two minutes.

## Brand Personality

Institutional · Reliable · Operational. The interface should convey that information entered here matters — that the system is a professional-grade tool in service of public safety. Not dramatic, not flashy. Calm precision with visible operational energy.

## Anti-references

Government legacy systems: the dense, outdated, form-heavy interfaces of older Colombian institutional software (SIAFI-era aesthetics, grey-background data grids, Comic Sans-adjacent typography, misaligned controls). NEXO-147 should feel modern and capable without crossing into consumer-app informality.

## Design Principles

1. **Clarity over chrome.** Every visual element must earn its place by reducing cognitive load or communicating status. Decoration that does not inform is a liability.
2. **Status is always visible.** Operators should never have to search for the current state of a report, a session, or an action. Feedback is immediate and unambiguous.
3. **Role shapes the interface.** Admin and operator surfaces have fundamentally different jobs. Don't let admin complexity leak into the operator's report form.
4. **Institutional weight.** The visual language signals that this is a serious, official system — not a startup prototype. Typography, spacing, and information density should match how professional tools look.
5. **Resilient in degraded conditions.** The interface should be legible and operable under poor lighting, on older monitors, and at smaller viewports without requiring a separate mobile build.

## Accessibility & Inclusion

Demo context — no strict WCAG level mandated. Body text contrast should meet AA (4.5:1) as a quality baseline, and all form inputs must be keyboard-accessible. No reduced-motion requirement, though existing animations should not block content.
